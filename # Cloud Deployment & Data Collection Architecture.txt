# cloud.md

# Cloud Deployment & Data Collection Architecture

## Project Context

This project is a weather forecast map web application.

The system will collect weather-related datasets from CWA OpenData APIs after the user provides the required API token.

The current project is a standalone prototype, but it must be designed so that it can later be merged into a larger project.

Future parent project:

* AI Agricultural Decision Intelligence Platform
* AI + Data Analytics + Interactive Dashboard
* Weather / Climate / Agriculture Decision Support

Therefore, all cloud, ETL, database, and deployment choices must avoid hard-coding and should remain modular.

---

# 1. Deployment Goal

## Prototype Goal

Build a low-cost, cloud-deployable prototype using mostly free-tier services.

Recommended prototype stack:

```text
Frontend:
- React / Next.js
- Vercel

Database:
- Supabase PostgreSQL
- PostGIS enabled
- JSONB support

ETL:
- Python scripts
- GitHub Actions Scheduler

Data Source:
- CWA OpenData API

Map:
- Windy API for map visualization only
```

The prototype should be easy to upgrade into:

```text

Database:
- PostgreSQL + PostGIS

- Docker worker

AI Layer:
- LLM Summary
- Weather Risk Interpretation
- Agriculture Decision Support
```

---

# 2. Architecture Principle

## Important Rule

Do not run crawler or scheduled ETL jobs in the frontend.

Frontend is only responsible for:

* UI
* Map display
* Dashboard visualization
* Calling backend/database API

ETL is responsible for:

* Fetching CWA JSON
* Validating data
* Normalizing nested JSON
* Saving data into PostgreSQL
* Preserving raw JSON when needed

# 5. Environment Variables

Create `.env.example` with the following variables:

```env
# CWA OpenData
CWA_API_TOKEN=

# Windy API
NEXT_PUBLIC_WINDY_API_KEY=

# Supabase
SUPABASE_URL=
SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
DATABASE_URL=

# Environment
APP_ENV=development
```

## Security Notes

* `CWA_API_TOKEN` must not be exposed in frontend.
* `SUPABASE_SERVICE_ROLE_KEY` must not be exposed in frontend.
* `DATABASE_URL` must only be used by ETL/backend.
* Only `NEXT_PUBLIC_WINDY_API_KEY` and `SUPABASE_ANON_KEY` may be used in frontend when necessary.
* Production security rules must be reviewed before public deployment.

# 6. Database Requirement

Recommended database:

```text
PostgreSQL + PostGIS + JSONB
```

Reason:

* Python ETL can easily write to PostgreSQL.
* CWA APIs return JSON.
* PostgreSQL supports JSONB for raw payload backup.
* PostGIS supports station location, township mapping, distance query, and future GIS analysis.
* This structure is compatible with future agriculture and climate analytics.

# 10. ETL Design Rules

Each ETL job should follow this pattern:

```text
Fetch CWA API
↓
Validate response
↓
Normalize JSON
↓
Upsert into PostgreSQL
↓
Save raw JSON into JSONB
↓
Log success or failure
```

Required ETL behavior:

* Use upsert instead of blind insert.
* Avoid duplicate records.
* Store raw API response in `raw_json`.
* Log failed API calls.
* Keep dataset-specific normalizers separated.
* Do not mix ETL logic with frontend logic.
* Do not hard-code tokens.
* Use environment variables.

# 11. Frontend Design Direction

Frontend should display:

* Windy map layer
* Weather station markers
* Rainfall station markers
* Township forecast layer
* Weather warning layer
* Typhoon layer
* Dashboard summary cards

Frontend should not directly fetch protected CWA APIs.

Frontend reads processed data from:

* Supabase read API, or
* future backend API

---

# 12. Windy API Role

Windy API is used for map visualization only.

It should not replace CWA OpenData as the official data source.

Use Windy API for:

* Base weather map
* Wind map
* Visual weather layer
* User interaction map

Use CWA OpenData for:

* Official observation data
* Official rainfall data
* Official township forecast
* Official warnings
* Official typhoon data
* Database storage
* Analytics
* AI summary

# 13. Merge Strategy for Future Parent Project

This weather map project will later be merged into a larger AI agriculture platform.

Therefore, please design the project with modular boundaries.

## Future merge target modules

```text
weather-map/
weather-etl/
weather-database/
weather-dashboard/
weather-ai-summary/
```

## Do not hard-code project-specific names

Avoid:

```text
app-only logic
single-use table names without namespace
frontend-dependent ETL
direct token usage in components
```

Prefer:

```text
modular ETL
clear data contracts
database schema files
environment variables
API abstraction layer
separate frontend and ETL folders
```
